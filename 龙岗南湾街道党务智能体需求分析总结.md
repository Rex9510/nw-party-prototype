# 龙岗南湾街道党务智能体 — 需求分析总结

> 基于《建设方案》《沟通需求记录》《访谈确认表》及沟通会议现场原型截图的深度交叉分析
> 最后更新：2026-04-25（含业务方确认结果）

---

## 一、项目概况

- **项目名称**：南湾红帆党务智管家
- **服务对象**：南湾街道全体党员（含流动党员）+ 党务工作者
- **用户规模**：4000人以内（龙岗区一个街道），前期试点约10余人
- **用户端形态**：党员端用微信小程序，管理员端用PC网站
- **核心定位**：集学习管理、服务保障、智能办公、教育培训于一体的党务智能助手
- **项目状态**：已立项，预算未最终确认
- **上线要求**：越快越好
- **运营人力**：约2人负责日常维护
- **部署方案**：需评估两套方案——阿里云 vs 本地物理服务器（含采购成本和维护成本对比）

## 三、已明确的功能需求（按模块梳理）— 全部为一期范围

### 模块一：流动党员智慧服务（小程序端）

#### 1. 个人基本信息展示
- 党员个人档案信息查看
- 历年参加活动情况汇总（按年份分组展示）
- 字段范围：**已确认**——姓名、编号、支部、流入时间、教育程度、手机号码，不需要其他字段
- **数据导入**：管理员通过Excel导入模板批量导入党员信息（姓名+手机号+身份证），导入后手机号即加入登录白名单

#### 2. 学习活动台账管理
- **三会一课/主题党日/志愿活动记录**：时间、地点、主题、内容 + 2张现场照片
- 频率：每月2-3场
- **线下活动签到流程**：
  - 每个活动生成唯一二维码，仅活动期间开放扫码
  - 党员现场扫码 → 进入小程序 → 微信授权登录 → 自动记录签到（即视为完成该次学习）
  - 活动结束后二维码关闭，不再接受扫码签到
  - 事后漏签只能由管理员手动补录
- **管理员操作**：后台录入上传党员参加活动情况，手动补录特殊情况（含纸质签字签到后的补录）

#### 3. 学习资源对接（全部通过 web-view 嵌入打开）
- 共产党员网 https://www.12371.cn
- 共产党员手机报（原型已有入口）
- 学习强国 https://www.xuexi.cn
- 深圳智慧党建 https://www.szzhdj.gov.cn/wxgzh/index.html#/learn/comprehensive?rand=1768385930293&tabId=411c1892c2194b81a8a9dfba72743842&isEnter=enterGzh
- **嵌入方式**：所有外部学习平台均通过小程序 web-view 嵌入打开，用户在小程序内完成学习
- **课程分类**：支持"必修"和"选修"区分（如二十届四中全会精神视频为必修集中观看），后台管理员可标记课程属性
- **学习时长统计**：
  - 通过 web-view 所在页面的 onShow/onHide 记录用户停留时长（精度90%以上，偏差秒级）
  - 自有课程/视频页面额外增加心跳机制（每30秒上报一次），提高统计精度
  - 外部H5平台（共产党员网等）仅依赖 onShow/onHide，无法加心跳
  - 统计维度：用户打开了哪个平台、停留时长、访问次数
  - 必须确保统计准确性

#### 4. 工作创业服务
- 岗位信息推送（就业信息入口）
- 惠企惠民政策解读、创业指导
- 投资简介手册（**已确认**：静态内容，后台链接配置管理）

#### 5. 生活便民服务
- 对接"i龙岗"小程序：`#小程序://i龙岗/wMeWysej0d7okgp`
- 整合日常生活场景小程序链接（闲鱼 `#小程序://闲鱼/ciBUlaoNGFEniDH`、外卖、咖啡、寄快递等）
- **管理方式：后台可配置，管理员可增删改**
- 配置项：图标 + 名称 + 链接类型 + 链接地址
- **前端交互参考**：参考"i龙岗"小程序的卡片式信息流布局（活动日历+图文卡片+分类标签横向滚动），整体风格简洁、图文并茂
- **跳转方案**（不做小程序关联）：
  - **H5链接**：小程序内通过 web-view 直接打开，无限制，优先使用此方式
  - **第三方小程序**：由于无法直接跳转未关联的小程序，采用以下方案：
    - 优先使用目标App的H5/移动网页版，通过 web-view 打开
    - 若无H5版本，页面展示小程序名称和图标，用户点击后弹窗提示"请在微信中搜索 XXX 小程序"，同时自动复制小程序名到剪贴板
    - 备选：通过公众号文章中转（web-view打开文章，文章内嵌小程序卡片，用户点击跳转）
- 旅游手册（**已确认**：静态内容，后台链接配置管理）

#### 6. 党务智能问答
- **实现方式：web-view 嵌入打开深小服H5页面**，不自建知识库
- 入口名称使用"党务智能问答"
- 深小服H5链接：https://apps.jinrirencai.com/vision-chat/index.html#/chat/ff908081965601960191460192688003
- 深小服的其他功能（每日一看/一学/一测、AI写作、党群地图等）：**仅跳转，不对标自建**

#### 7. AI智能画像
- **功能定义**：系统自动总结个人学习情况、参会情况、需要补足的学习情况等
- **受众**：党员自己（自我提醒学习参会）、管理员、上级
- **应用场景**：可用于评优评先、年度总结等管理场景
- **设计目标**：画像内容要让党员和管理员认可度高、接受度高，体现足够的智能化水平
- 基于本系统内活动参加情况进行多维分析

##### AI画像统计口径与指标体系（V1）

**一、数据采集维度**

| 数据源 | 采集内容 | 采集方式 |
|--------|----------|----------|
| 线下活动签到 | 三会一课、主题党日、志愿活动的出勤记录 | 扫码签到 + 管理员补录 |
| 直播课程签到 | 直播培训的参与记录 | 签到链接登录记录 |
| 录播课程观看 | 课程观看记录、观看时长、完成率 | 系统内播放记录 |
| 外部学习平台 | 共产党员网/学习强国/智慧党建的访问次数和停留时长 | web-view onShow/onHide |
| 自有课程学习 | 课程学习时长、完成进度 | 心跳机制上报 |
| 必修课程完成 | 必修课程是否全部完成 | 课程属性标记 + 学习记录 |

**二、核心指标体系**

| 指标名称 | 计算口径 | 权重建议 |
|----------|----------|----------|
| 活动出勤率 | 实际参加活动次数 ÷ 应参加活动总次数 × 100% | 25% |
| 必修课程完成率 | 已完成必修课程数 ÷ 必修课程总数 × 100% | 20% |
| 学习总时长 | 所有学习平台累计停留时长（小时） | 15% |
| 直播培训参与率 | 实际参加直播次数 ÷ 直播总场次 × 100% | 15% |
| 录播课程学习量 | 已观看录播课程数（完成率≥80%视为完成） | 10% |
| 学习主动性 | 非必修课程的自主学习次数和时长 | 10% |
| 学习持续性 | 连续活跃周数（每周至少1次学习或活动记录） | 5% |

**三、画像输出内容**

1. **综合评分**：基于上述指标加权计算，满分100分，划分等级：
   - 优秀（90-100）、良好（75-89）、合格（60-74）、待提升（<60）

2. **个人总结文案**（AI自动生成，自然语言描述）：
   - 学习亮点：本周期内表现突出的方面（如"连续12周保持学习活跃"）
   - 参会概况：出勤情况总结（如"本季度参加三会一课8次，出勤率100%"）
   - 待补足项：未完成的必修课程、缺席的活动、学习时长不足等
   - 成长趋势：与上一周期对比的变化（如"学习时长环比增长30%"）
   - 个性化建议：针对薄弱项给出具体行动建议

3. **可视化图表**：
   - 雷达图：多维度能力展示（出勤、学习、培训、主动性、持续性）
   - 趋势折线图：按月展示综合评分变化
   - 排名参考：在所属支部中的排名百分位（如"超过85%的党员"）

4. **管理员/上级视角额外内容**：
   - 支部整体画像：全员平均分、各等级人数分布
   - 重点关注名单：连续2个周期"待提升"的党员
   - 评优推荐名单：连续保持"优秀"等级的党员，可直接用于评优评先参考

**四、分析周期与触发规则**

| 周期 | 生成时间 | 内容侧重 |
|------|----------|----------|
| 周报 | 每周一凌晨自动生成 | 本周学习/活动参与概况 + 下周待完成提醒 |
| 月报 | 每月1日凌晨自动生成 | 月度综合评分 + 环比变化 + 待补足项 |
| 年报 | 每年1月1日自动生成 | 年度总结 + 成长轨迹 + 评优推荐 |
| 主动触发 | 党员随时点击 | 截至当前的实时画像 |
- **自动分析周期**：周自动分析、月自动分析、年自动分析，定时自动生成
- **主动触发**：个人也可以随时点击触发AI画像更新生成
- **自动提醒机制**：通过微信小程序订阅消息推送，用户首次登录时引导开启订阅权限
- **未读二次推送**：推送后记录状态，用户点击进入小程序时标记"已查看"，对未查看用户定时二次推送（注：微信不提供已读回调，通过用户是否点击进入来判断）
- **管理员视角**：后台可直观查看每个党员的学习情况、参会情况汇总

#### 8. 二维码功能
- 扫码进入小程序
- 线下参会签到扫码
- 线上参会/学习签到扫码
- 签到行为记录

### 模块二：智能党务工作工具（PC管理端）

#### 1. 党员档案智能审核（沟通记录中称"资料智能检查"）
- **审核标准知识库**：后台管理员上传审核标准文档，标准会不断更新迭代，系统需支持版本管理
- **支持上传格式**：图片、PDF、Word等常用文档形式、电子档
- **实现方式**：类似 RAG 方案——AI 基于管理员上传的审核标准知识库，对本地上传的待审查文件/图片进行智能识别与比对
- **产出物**：生成审核结果（Word格式导出）+ 审查建议修改方向，供人工复核
- 档案信息自动校验、完整性核查
- 问题条目智能标注

#### 2. 不合格党员管理与处置指导
- 情形认定参考、流程规范指引
- 材料模板、处置步骤、政策依据智能推送
- **注意**：系统仅提供参考指引，不自动作出组织处理结论

### 模块三：街道党校智能培训

#### 1. 课程资源中心（录播）
- AI办公工具教学课程
- 党务工作、政策理论、数字技能等课程
- 支持后台上传管理课程内容
- **学习统计**：自动统计每个录播课的观看人数及个人观看记录

#### 2. 直播课程服务（独立入口）
- **直播平台**：第三方SaaS（示例：https://live.avalive.cn），每月2-3场，最多约4000人同时在线
- **直播文案生成**：管理员后台创建直播活动后，系统自动生成含签到链接+二维码的分享文案（示例格式：活动名称、时间、讲师、直播链接、签到二维码、保密提示等）
- **签到流程**：
  - 签到链接在直播开始前30分钟开放
  - 党员点击链接 → 进入小程序 → 登录（微信授权手机号 或 手机号+验证码，二选一）→ 自动记录参与人姓名（即完成签到）→ 跳转第三方直播页面观看
  - PC端：打开链接 → 手机号+验证码登录 → 记录参与 → 跳转第三方直播页面
  - 系统自动去重，同一人重复签到只记一次
- **留档**：直播录制留档（由第三方平台提供回放）
- **学习记录**：签到即记录参与，写入个人学习记录表
- **AI总结**：基于参与记录，AI自动总结每个人的直播学习情况
- 精品直播课预告、回看
- 每月优质课程预告发布
- 参训记录同步记录

### 模块四：后台管理系统（PC端）

- **角色权限管理**：系统设计角色权限体系（如超级管理员、社区管理员、审计员等），权限控制到页面和按钮级别；审计员可查看操作日志和问答日志
- 党员活动信息录入与上传
- 课程内容上传与管理
- 生活服务链接管理（图标+链接类型+地址，可增删改）
- **党员学习/参会统计报表**：管理员可直观查看每个党员的学习情况、参会情况、观看记录等汇总数据
- 信息统计与查阅
- 特殊场景编辑
- 每个社区1-2个管理员

| 部署方案 | 需评估两套：阿里云 vs 本地物理服务器 |
| 数据安全 | 党员个人信息高度保护：个人信息、年龄、地址、电话、单位、党龄等字段均不可泄露，不可进入AI问答上下文，架构设计需严格保障隐私 |
| 大模型合规 | 使用 DeepSeek V4 Flash 普通 API；注意：业务方已接受普通 API 方案，原"不允许问答日志提交厂商用于模型改进"的要求不再作为硬性约束 |
| 数据保留 | 永久保留，状态机变更记录+时间记录，支持人工删除 |
| 档案审核 | RAG方案：管理员上传审核标准知识库（支持版本更新），待审查文件支持图片/PDF/Word上传，AI基于标准知识库识别比对后生成审核结果+修改建议（Word导出），供人工复核 |
| 直播方案 | 第三方SaaS（如avalive），签到链接+二维码，登录即签到，自动去重，跳转第三方观看 |
| 验收标准 | 功能+性能+安全测评 |
| 等保与安全评估 | 需评估网络安全等级保护要求（政务系统通常要求等保二级或以上），是否需要商用密码应用安全性评估，具体方案由技术侧出 |
| 权限与审计 | 系统需设计角色权限体系，控制到后台页面和按钮级别；操作日志和问答日志需支持审计角色查看，角色权限按需分配 |
| 运维模式 | 允许远程登录生产环境，可短时间驻场；建议：日常远程运维为主，上线初期及重大更新时短期驻场（1-2周），配合2人运营团队做知识转移 |

## 六、需求矛盾与风险点

1. **学习时长统计准确性**：外部学习平台（共产党员网、学习强国、深圳智慧党建）通过 web-view 嵌入打开，依赖 onShow/onHide 统计停留时长，精度约90%，存在锁屏延迟、进程被杀等边界情况导致的偏差。自有课程通过心跳机制可提高精度。需在产品层面明确：这个精度是否满足业务对"准确性"的要求。

2. **AI画像提醒的微信订阅消息限制**：微信小程序订阅消息分为一次性订阅和长期订阅。长期订阅仅对政务民生、金融等特定类目开放，党务类小程序需确认是否符合长期订阅资质。若不符合，每次推送前都需要用户手动点击授权，体验较差。首次登录引导订阅可以缓解，但无法完全解决。

3. **不合格党员处置功能的敏感性**：该功能涉及组织纪律，系统边界必须严格限定在"参考指引"层面，需法务/组织部门确认。

4. **4000用户但试点仅10余人**：需要明确从试点到全面推广的路径和时间表。

5. **直播4000人并发**：腾讯会议企业版需要评估对应的许可证费用，4000人同时在线对SaaS方案有成本影响。

6. **双部署方案评估**：阿里云和本地服务器两套方案的成本对比需要尽早完成，影响整体预算。

7. **大模型方案已确认**：使用 DeepSeek V4 Flash 普通 API，成本低、接入快。需注意：API 调用时仍应避免将党员敏感字段（身份证、地址、电话等）直接传入 prompt，在应用层做脱敏处理即可。

8. **档案智能审核的AI能力要求**：采用 RAG 方案，需要：OCR 识别图片/扫描件 + 文档解析（PDF/Word）+ 审核标准知识库的向量化与检索 + 内容比对校验逻辑。额外风险：审核标准会不断更新，需要设计知识库的增量更新与索引刷新策略，避免旧标准残留导致审核结果不准确。

9. **一期范围全上的工期风险**：所有功能均确认为一期，其中AI画像、档案智能审核、直播集成+自动学习总结三个模块复杂度高，加上要出两套部署方案，需要评估团队资源是否能支撑"越快越好"的上线要求。建议识别出可以先灰度上线的最小可用版本。

10. **第三方小程序跳转限制**：不做小程序关联，无法直接跳转闲鱼、外卖等第三方小程序。解决方案：优先使用目标App的H5版本通过web-view打开；无H5版本的，引导用户搜索目标小程序（自动复制名称到剪贴板）；或通过公众号文章内嵌小程序卡片中转。需要逐个确认目标服务是否有可用的H5版本。

## 八、数据与权限设计

### 8.1 核心数据对象（V1）

| 数据对象 | 关键字段 |
|----------|----------|
| 党员档案 | 党员唯一ID、姓名、所属支部、流入时间、联系方式、教育程度、状态（在册/流动/转出等） |
| 活动台账 | 活动ID、活动类型（三会一课/主题党日/志愿活动）、时间、地点、组织单位、签到规则、照片材料 |
| 学习记录 | 学习记录ID、学习资源类型（录播/直播/外部平台）、开始/结束时间、时长、完成度、来源端（小程序/PC） |
| 签到记录 | 签到ID、活动ID/直播ID、签到人、签到方式（扫码/补录/链接登录）、签到时间、去重标识 |
| 画像结果 | 画像ID、周期类型（周/月/年/实时）、指标明细、综合评分、等级、AI总结文案 |
| 审核任务 | 任务ID、审核标准版本、待审文件、问题条目、审核结论、导出报告 |
| 操作日志 | 操作人、角色、操作对象、操作类型、时间、IP/设备信息、结果状态 |

### 8.2 数据分级与隐私原则（V1）

- **高敏数据**：党员身份信息（身份证号）、联系方式（手机号、地址）、组织关系、处置相关材料
- **处理原则**：
  - 最小必要：仅采集业务必需字段
  - 按角色可见：不同角色看到的数据范围严格隔离
  - 默认不外发：高敏数据不出系统边界
  - 全链路留痕：所有数据访问和变更均记录审计日志
- **AI调用约束**：涉及大模型调用时，对高敏字段先脱敏处理；身份证号、电话、详细住址等禁止直接拼入 Prompt
- **数据保留**：业务数据默认长期保存；删除操作需人工审批并留审计记录

### 8.3 权限矩阵（V1）

| 角色 | 党员档案 | 活动台账 | 学习/画像 | 审核任务 | 系统配置 | 日志审计 |

| 社区管理员 | 查看本社区 | 新增/编辑/补录本社区 | 查看本社区统计 | 提交审核任务 | 配置本社区基础内容 | 查看本社区日志 |
| 街道管理员 | 全街道查看 | 全街道管理 | 全街道统计/导出 | 审核与结果导出 | 全局配置 | 查看全局日志 |

## 九、核心流程异常处理补充

> 注：主流程已在第三节各模块中详细描述，此处仅补充异常场景的处理规则。

### 9.1 活动签到异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 重复扫码 | 仅保留首条有效签到记录，后续扫码静默忽略 |
| 活动改期/取消 | 活动状态变更同步到签到页面，已签到记录保留并标记原活动状态 |
| 网络异常 | 客户端本地暂存签到请求，恢复后自动重试提交（带幂等ID防重复写入） |
| 二维码截图转发 | 签到时校验微信授权登录身份，截图转发无法完成授权，不影响安全性 |

### 9.2 直播培训异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 签到链接过期 | 提示"活动已结束"，不再接受签到 |
| 同一用户多端登录 | 参与记录按用户ID去重，仅记一次 |
| 第三方直播平台不可用 | 展示备用通知页面，告知用户稍后查看回放；管理员可手动补录参与记录 |

### 9.3 档案智能审核异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 上传文件格式不支持 | 前端校验+后端兜底，提示支持的格式列表（图片/PDF/Word） |
| OCR识别失败或置信度低 | 标注"识别不确定"条目，提醒人工复核 |
| 审核标准知识库更新后旧任务 | 已完成的审核任务绑定当时的标准版本号，不受后续更新影响；支持基于新标准重新审核 |

> 边界说明：系统仅提供辅助审核建议，不替代组织最终决定。
